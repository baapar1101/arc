# سناریو: انتقال انفجار فرمول از بخش کالا به فاکتور تولید

## وضعیت فعلی

### 1. مدیریت فرمول تولید (BOM) در بخش کالا
- **مکان**: تب "فرمول تولید" در فرم ویرایش/ایجاد کالا (`product_form_dialog.dart`)
- **عملکرد فعلی**:
  - کاربر می‌تواند فرمول‌های تولید را برای هر کالا تعریف کند
  - فرمول‌ها در جدول `product_boms` ذخیره می‌شوند و به کالا لینک می‌شوند (`product_id`)
  - هر فرمول می‌تواند چندین نسخه داشته باشد (`version`)
  - یک فرمول می‌تواند به عنوان پیش‌فرض تنظیم شود (`is_default`)
  - در همین بخش، دکمه "انفجار فرمول" وجود دارد که:
    - مقدار تولید را از کاربر می‌گیرد
    - فرمول را منفجر می‌کند (`explode_bom` API)
    - نتایج را در یک دیالوگ نمایش می‌دهد
    - امکان ایجاد "پیش‌نویس سند تولید" را می‌دهد (نه فاکتور!)

### 2. صدور فاکتور تولید
- **مکان**: صفحه `new_invoice_page.dart`
- **نوع فاکتور**: `InvoiceType.production` (مقدار API: `invoice_production`)
- **عملکرد فعلی**:
  - کاربر می‌تواند فاکتور از نوع "تولید" را ایجاد کند
  - باید به صورت دستی ردیف‌های فاکتور را اضافه کند
  - هیچ ارتباطی با فرمول‌های تولید ندارد

## مشکل فعلی
- انفجار فرمول در بخش کالا انجام می‌شود و یک "سند" ایجاد می‌کند، نه "فاکتور"
- برای صدور فاکتور تولید، کاربر باید به صورت دستی تمام مواد اولیه و خروجی‌ها را وارد کند
- هیچ راه مستقیمی برای استفاده از فرمول‌های ذخیره‌شده در فاکتور تولید وجود ندارد

## سناریو پیشنهادی

### هدف
1. **حفظ مدیریت فرمول در بخش کالا**: فرمول‌ها همچنان در تب "فرمول تولید" کالا تعریف و مدیریت می‌شوند
2. **انتقال انفجار به فاکتور**: انفجار فرمول باید در صفحه صدور فاکتور تولید انجام شود

### تغییرات پیشنهادی

#### 1. بخش کالا (Frontend - `product_bom_section.dart`)
**تغییرات**:
- ✅ **حفظ**: مدیریت فرمول‌ها (ایجاد، ویرایش، حذف) همچنان در این بخش باقی می‌ماند
- ❌ **حذف**: دکمه "انفجار فرمول" از این بخش حذف می‌شود
- ❌ **حذف**: دیالوگ نمایش نتایج انفجار و ایجاد پیش‌نویس سند حذف می‌شود
- ✅ **افزودن**: یک پیام راهنما یا لینک به کاربر که "برای استفاده از این فرمول، در فاکتور تولید از آن استفاده کنید"

#### 2. صفحه صدور فاکتور تولید (Frontend - `new_invoice_page.dart`)
**تغییرات**:
- ✅ **افزودن**: یک دکمه/ویجت جدید در تب "کالاها و خدمات" که فقط برای فاکتورهای تولید نمایش داده می‌شود
- ✅ **ویجت انفجار فرمول**:
  - انتخاب کالای تولیدی (از لیست کالاها)
  - نمایش فرمول‌های موجود برای آن کالا (اگر فرمول پیش‌فرض وجود دارد، به صورت پیش‌فرض انتخاب شود)
  - امکان انتخاب فرمول خاص (اگر چند فرمول وجود دارد)
  - ورود مقدار تولید
  - دکمه "انفجار و افزودن به فاکتور"
- ✅ **پس از انفجار**:
  - نتایج انفجار به صورت ردیف‌های فاکتور به `_lineItems` اضافه می‌شوند
  - مواد اولیه (inputs) با `movement: "out"` به عنوان مصرف مواد
  - خروجی‌ها (outputs) با `movement: "in"` به عنوان محصول تولید شده
  - کاربر می‌تواند قبل از ذخیره، ردیف‌ها را ویرایش کند

#### 3. Backend (بدون تغییر)
- ✅ API `explode_bom` همچنان موجود است و استفاده می‌شود
- ✅ API `produce_draft` می‌تواند حذف شود یا برای استفاده‌های دیگر نگه داشته شود
- ✅ ساختار دیتابیس بدون تغییر باقی می‌ماند

### جریان کار پیشنهادی

```
1. کاربر در بخش کالا:
   └─> کالای تولیدی را ایجاد/ویرایش می‌کند
   └─> در تب "فرمول تولید"، فرمول‌های تولید را تعریف می‌کند
   └─> فرمول را ذخیره می‌کند

2. کاربر برای صدور فاکتور تولید:
   └─> به صفحه "صدور فاکتور جدید" می‌رود
   └─> نوع فاکتور را "تولید" انتخاب می‌کند
   └─> به تب "کالاها و خدمات" می‌رود
   └─> دکمه "انفجار فرمول" را می‌زند
   └─> کالای تولیدی را انتخاب می‌کند
   └─> فرمول (یا فرمول پیش‌فرض) انتخاب می‌شود
   └─> مقدار تولید را وارد می‌کند
   └─> دکمه "انفجار و افزودن" را می‌زند
   └─> ردیف‌های فاکتور به صورت خودکار پر می‌شوند:
       ├─ مواد اولیه (movement: out)
       └─ محصولات تولید شده (movement: in)
   └─> کاربر می‌تواند ردیف‌ها را ویرایش کند (مقدار، قیمت، انبار، ...)
   └─> فاکتور را ذخیره می‌کند
```

### جزئیات پیاده‌سازی

#### Frontend - ویجت انفجار فرمول در فاکتور

**فایل جدید**: `hesabixUI/hesabix_ui/lib/widgets/invoice/bom_explosion_widget.dart`

```dart
class BomExplosionWidget extends StatefulWidget {
  final int businessId;
  final Function(List<InvoiceLineItem>) onExploded;
  
  // ...
}
```

**عملکرد**:
1. نمایش یک دکمه/کارت در تب "کالاها و خدمات" (فقط برای `InvoiceType.production`)
2. دیالوگ انتخاب کالا و فرمول
3. ورود مقدار تولید
4. فراخوانی API `explode_bom`
5. تبدیل نتایج به `InvoiceLineItem` و افزودن به لیست

**تغییرات در `new_invoice_page.dart`**:
- در متد `_buildProductsTab()`، اگر `_selectedInvoiceType == InvoiceType.production`:
  - ویجت `BomExplosionWidget` را نمایش بده
  - پس از انفجار، ردیف‌ها را به `_lineItems` اضافه کن

#### Frontend - حذف انفجار از بخش کالا

**تغییرات در `product_bom_section.dart`**:
- حذف متد `_explode()`
- حذف دکمه "انفجار فرمول" از `ListTile`
- (اختیاری) افزودن یک `InfoCard` که راهنمایی می‌دهد: "برای استفاده از این فرمول، در فاکتور تولید از آن استفاده کنید"

### مزایای این رویکرد

1. ✅ **جداسازی مسئولیت‌ها**: 
   - مدیریت فرمول = بخش کالا
   - استفاده از فرمول = فاکتور تولید

2. ✅ **تجربه کاربری بهتر**:
   - کاربر مستقیماً در فاکتور می‌تواند از فرمول استفاده کند
   - نیازی به رفتن به بخش کالا و بازگشت نیست

3. ✅ **انعطاف‌پذیری**:
   - کاربر می‌تواند پس از انفجار، ردیف‌ها را ویرایش کند
   - می‌تواند ردیف‌های اضافی دستی هم اضافه کند

4. ✅ **سازگاری با Backend**:
   - از API‌های موجود استفاده می‌کند
   - نیاز به تغییر Backend نیست

### سوالات برای بررسی

1. آیا می‌خواهید امکان ویرایش ردیف‌های انفجار شده قبل از ذخیره وجود داشته باشد؟ ✅ (پیشنهاد: بله)

2. آیا می‌خواهید امکان انفجار چندباره (برای کالاهای مختلف) در یک فاکتور وجود داشته باشد؟ ✅ (پیشنهاد: بله)

3. آیا می‌خواهید اگر کاربر ردیف‌های دستی اضافه کرده، بتواند بعداً هم انفجار انجام دهد؟ ✅ (پیشنهاد: بله)

4. آیا می‌خواهید اطلاعات فرمول (مثل BOM ID) در `extra_info` ردیف‌های فاکتور ذخیره شود؟ (برای ردیابی)

5. آیا می‌خواهید امکان "حذف ردیف‌های انفجار شده" به صورت گروهی وجود داشته باشد؟
